# Multi-Tenant SaaS CRM B2B - Project Instructions

## Project Overview

**Sales CRM** is a Laravel 12-based multi-tenant B2B SaaS application for managing sales pipelines. Each tenant (company) has isolated data with role-based access control for Admin, Manager, and Agent roles.

### Key Architecture
- **Multi-Tenancy**: Database-per-tenant approach with `tenant_id` column on all core tables
- **RBAC**: Spatie Laravel Permission for role-based access control
- **API**: Laravel Sanctum for token authentication
- **Frontend**: Blade templates + Alpine.js + Tailwind CSS (CDN)
- **i18n**: English (EN) and Arabic (AR) with RTL support

## Multi-Tenancy Architecture

### Tenant Isolation
1. **Global Scope**: All models with `tenant_id` have an Eloquent global scope that filters by current tenant
2. **Tenant Model**: `App\Models\Tenant` manages subscription plans, trial periods, and subdomains
3. **Tenant Detection**: Via `auth()->user()->tenant_id` (called from middleware)

### Tables with Tenant Isolation
- `users` (tenant_id FK)
- `contacts` (tenant_id FK)
- `deals` (tenant_id FK)
- `activities` (tenant_id FK)

### Example Global Scope (Already Implemented in Deal.php)
```php
protected static function booted(): void
{
    static::addGlobalScope('tenant', function ($query) {
        $tenantId = auth()->user()?->tenant_id;
        if ($tenantId) {
            $query->where('tenant_id', $tenantId);
        }
    });
}
```

**ACTION**: Apply this same pattern to `Contact`, `Activity`, and `User` models if not already done.

## Core Data Model

### Tenant
- `id`, `name`, `subdomain`, `plan` (free/pro/enterprise), `is_active`, `trial_ends_at`
- Relations: `hasMany(User)`, `hasMany(Contact)`, `hasMany(Deal)`, `hasMany(Activity)`

### User
- `id`, `email`, `password`, `tenant_id` (FK), `name`, `phone`
- Roles: Admin, Manager, Agent
- Relations: `belongsTo(Tenant)`, `hasMany(Deal)`, `hasMany(Activity)`

### Contact
- `id`, `tenant_id` (FK), `user_id` (FK - owner), `name`, `email`, `company`, `phone`, `status`, `source`
- `tags` (JSON), `custom_fields` (JSON)
- Relations: `belongsTo(Tenant)`, `belongsTo(User)`, `hasMany(Deal)`, `hasMany(Activity)`

### Deal
- `id`, `tenant_id` (FK), `contact_id` (FK), `user_id` (FK), `title`, `value` (decimal), `currency`, `stage`, `probability` (0-100), `expected_close_date`, `lost_reason`
- Stages (Constants): `New`, `Contacted`, `Qualified`, `Proposal`, `Negotiation`, `Won`, `Lost`
- Relations: `belongsTo(Tenant)`, `belongsTo(Contact)`, `belongsTo(User)`, `hasMany(Activity)`

### Activity
- `id`, `tenant_id` (FK), `user_id` (FK), `contact_id` (FK, nullable), `deal_id` (FK, nullable), `note`, `type`, `due_date`, `is_done`
- Types: `Call`, `Meeting`, `Email`, `Task`, `Demo`
- Relations: `belongsTo(Tenant)`, `belongsTo(User)`, `belongsTo(Contact)`, `belongsTo(Deal)`

## Routes & Role-Based Access

### Route Structure
All routes must use role-specific prefixes:
```php
route('admin.deals.index')      // Admin routes
route('manager.deals.index')    // Manager routes
route('agent.contacts.create')  // Agent routes
```

### Route Prefixes (see routes/web.php)
```php
// Admin group
Route::prefix('admin')->middleware(['auth:sanctum', 'role:Admin'])->name('admin.')->group(...)

// Manager group
Route::prefix('manager')->middleware(['auth:sanctum', 'role:Manager'])->name('manager.')->group(...)

// Agent group
Route::prefix('agent')->middleware(['auth:sanctum', 'role:Agent'])->name('agent.')->group(...)
```

### Authorization Template
```php
private function authorizeContact(Contact $contact): void
{
    $user = auth()->user();
    if (!$user->hasRole(['Admin', 'Manager']) && $contact->user_id !== $user->id) {
        abort(403, 'Unauthorized action.');
    }
}
```

## Security Guidelines (CRITICAL)

1. **SQL Injection**: Always use query builder with parameter binding - NEVER string concatenation
2. **XSS**: Use `{{ }}` for user data (escaped), `{!! !!}` ONLY for trusted content
3. **CSRF**: All POST forms must have `@csrf` directive
4. **Mass Assignment**: Always define `$fillable` on models
5. **Authorization**: Use role checks and custom authorization methods
6. **Rate Limiting**: Use `throttle:attempts,minutes` middleware
7. **External Links**: Add `rel="noopener noreferrer"` to `target="_blank"` links
8. **Tenant Isolation**: Always verify user's `tenant_id` doesn't access other tenants
9. **Carbon Date Handling**: Use `$date->copy()` to avoid mutations - never chain mutations

## Database Performance

### N+1 Prevention (CRITICAL)
Always eager load relationships in queries:
```php
// ‚úÖ GOOD
Deal::with(['contact', 'user', 'tenant'])->paginate();
Activity::with(['deal', 'user', 'contact'])->get();

// ‚ùå BAD - N+1 query problem
Deal::all()->each(fn($d) => $d->contact->name);
```

### Indexes
- `tenant_id` indexed on all core tables
- `user_id` indexed for user-owned data
- `contact_id` indexed for deal/activity queries
- `stage` indexed on deals (for Kanban filtering)

## Common Issues & Fixes

### N+1 Query Problem
**Symptom**: Slow page loads despite low record count
**Fix**: Use eager loading with `with()`: `Deal::with('contact', 'user')->paginate()`

### Tenant Data Leaking
**Symptom**: User sees another tenant's deals
**Fix**: Ensure global scope is added to model. Check: `Deal::with('contact')->get();` filters by tenant_id

### Carbon Mutation Bugs
**Symptom**: Date calculations accumulate incorrectly
**Fix**: Use `$date->copy()->addDay()`, never chain mutations: `now()->addDay()->addDay()`

### Missing Translations
**Symptom**: `messages.key` appears in UI instead of translation
**Fix**: Add key to lang/en/messages.php and lang/ar/messages.php

### CSRF Token Errors
**Symptom**: POST requests return 419 error
**Fix**: Add `@csrf` directive to all forms in Blade templates

## Performance Optimization

- Use pagination for large datasets: `.paginate(15)`
- Index frequently-queried columns (`tenant_id`, `user_id`, `stage`)
- Cache expensive queries in services
- Use queue jobs for heavy operations (SendActivityReminders)
- Lazy-load properties where appropriate
- Monitor with Laravel Telescope (dev only)
